Cascading Style Sheets (CSS)

Learn CSS

  1. Tells the browser how to display a certain element
  2. Offers lots of power & versatility: CSS Zen Garden
  3. Follows the general ruleset:
    1. Select the HTML element to be styled
    2. Specify the properties of the element to be styled
    3. Give the values we want each property to have
p {
    color: orange
    font-size: 16px;
}
Not a programming language like Javascript, nor a mark-up language like HTML
Nothing quite like it
You can think of it like a Style Configuration File
Easy to get started using & understand but very hard to master

Structure of a CSS rule

selector {
  property1: value;
  property2: value;
  property3: value;
  property4: value;
}
a {
  color: #c929af;
  font-size: 24px;
  padding: 8px;
  background-color: gold;
}

Example <a> tag

  • The selector identifies which HTML element(s) the style rule will be applied to
  • The curly braces contain the property-value pairs, each separated by a semi-colon
  • The property indicates the style to be changed
  • The value is what the properties are set to for that element

One of the most complicated parts about CSS is specificity

  • Elements, classes and ids have increasing degrees of specificity

  • Combining Selectors increases specificity

  • Highest specificity wins. Order also matters. !important doesn't change specificity, but it does take precedence over all.

  • In general, you want to choose selectors with smallest possible specificity, so you do have to fight it later

--name: "value";

--my-font: Montserrat, sans-serif;
}

Invoke a property with var():

button {

}

MDN: Custom Properties

CSS Variable Scoping

  • The variables on the :root selector will be available to all elements on the page
  • But you can declare variables lower down in the HTML render tree by using more specific selectors
  • This will "scope" those variables to only particular parts of your site

:root {
--my-color: royalblue;    
}
footer {
--my-color: darkorchid;
}
body, footer {
color: var(--my-color)
}
<body>
<p>This is body text</p>
<footer>This is my footer</footer>
</body>

CSS Math Functions

calc() min() max() clamp()

CSS supports dynamic calculations:

div {
  width: calc(100% - 20px);
  height: max(100%, 400px);
  margin: min(16px, 1vw);
  font-size: clamp(12px, 2vmin, 28px);

  /* combine them! */ 
  padding: calc(4px * min(calc(50% / 2), 20vh));

  /* you can use /, *, -, or + operators with calc(), just
  don't forget the spaces and match up all the parens! */
}

How browsers render stuff

visualized
  • Different browsers use different rendering engines but follow similar flow
  • Browser will parse the markup, then "Recalculate Style" to figure out what styles apply to which elements
  • Next, it will "Layout" all the DOM elements
  • Then, "Paint" the pixel data for the elements
  • And, "Composite" will combine and render the painted layers out onto the screen

Rendering: HTML and CSS Parsers

Browser creates the DOM and the CSSOM

The DOM

The CSSOM

The Render Tree

But if you change anything in the DOM or CSSOM, the Render Tree must be recalculated

Think of it as a pyramid

If you change something at the bottom, the whole pyramid has to be rebuilt.

If you change just the top layer, the foundation beneath remains intact.

Layout

  • Generate geometry and position of each element
  • Very expensive because engine must reflow the relevant element and all its children to take into account the new layout
  • Occurs when the DOM tree is manipulated, or a style which affects layout is changed, when browser window size is changed, when className property is changed

Paint

  • Fill out pixels for each element into layers
  • Make something visible when it previously was not, like adding outline or changing background colour
  • Expensive because the engine searches through all elements to determine what is visible and what should be displayed
  • Browser vendors are scrambling to improve the renderer, but until recently browsers weren't built to handle animations efficiently

Composite

These are safe to dynamically change and animate.

  • Opacity and Transforms trigger composite
  • Blink and Webkit creates new layer for elements with CSS transforms or opacity
  • translateZ(0) is used to force layer creation which ensures both the layer is painted as soon as the animation starts
  • Watch The Web's Black Magic for best explanation on rendering and compositing

Why is compositing so much faster?

  • GPU accelerated compositing - browser developers started moving things over to the GPU.
  • opacity, transform, filter, clip-path can be GPU-accelerated
  • There are two tasks that could potentially move over to the GPU:
    • Painting the layers - harder to do
    • Compositing them together - easier to do

Using the GPU for the paint step is supported on some browsers under some conditions, but not widespread yet. But they're working on it!

Bottom Line

Reach for these first when animating:

  • transform
    ...ignores inline elements (i.e. <span>); change to inline-block/block
  • opacity
  • filter
  • clip path & background-color are soon to be added (in chrome)
  • If you are animating a property (especially one above), will-change MDN creates a composite layer for it
  • i.e. Animating color? will-change: color;
